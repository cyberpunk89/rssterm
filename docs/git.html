Basics

Git is a tool that takes snapshots of your files over time, so you can see exactly what changed, go back to any previous version, and collaborate with others without overwriting each other's work. Every project is a repository (repo) — a folder Git is watching.

The three areas

Before any command makes sense, understand where your files live at any moment:

git areas
┌─────────────────┐ git add ┌─────────────────┐ git commit ┌─────────────────┐ │ Working Dir │ ─────────────▶ │ Staging Area │ ─────────────▶ │ Repository │ │ │ │ │ │ │ │ Your files │ │ Changes you │ │ Saved history │ │ as you edit │ │ chose to save │ │ (commits) │ └─────────────────┘ └─────────────────┘ └─────────────────┘ # Edit a file → it's in the Working Dir # git add → it moves to Staging # git commit → it's permanently saved in the Repo

First-time setup

Run these once. Git uses your name and email to label every commit you make.

bash
git config --global user.name  "Your Name"
git config --global user.email "you@example.com"

# Verify it was saved
git config --list
💡 Tip

Set your default branch name to main to match GitHub's default: git config --global init.defaultBranch main

Starting a project

git init — track an existing folder

bash
$ mkdir my-project && cd my-project
$ git init
Initialized empty Git repository in /my-project/.git/

That's it — Git is now watching this folder. The .git/ folder it creates is where all the history lives. Don't touch it.

git clone — copy an existing repo

Use this when a project already exists (on GitHub or elsewhere).

bash
git clone https://github.com/user/repo.git

# Clone into a specific folder name
git clone https://github.com/user/repo.git my-folder

.gitignore — tell Git what to skip

Create a .gitignore file in the root of your project. Any file or pattern listed there will be completely ignored by Git — it won't even show up in git status.

.gitignore
# Dependencies
node_modules/
vendor/

# Build output
dist/
build/
*.o
*.class

# Environment / secrets
.env
.env.local
*.pem
secrets.yaml

# OS files
.DS_Store
Thumbs.db

# Editor
.vscode/
.idea/
*.swp
💡 Tip

Never commit secrets. If you accidentally committed a password or API key, change it immediately — even if you delete the file later, the key is still in Git history.

Daily workflow

These four commands are what you'll use every single day. Learn them cold.

git status — see what's going on

Always start here. It shows you what's changed, what's staged, and what's untracked.

bash
$ git status

On branch main
Changes not staged for commit:
  modified:   src/app.tsx

Untracked files:
  new-file.ts

# Short version — one line per file
$ git status -s

git add — choose what to save

bash
# Stage a specific file
git add src/app.tsx

# Stage everything in the current directory
git add .

# Stage parts of a file interactively
git add -p src/app.tsx

git commit — save the snapshot

bash
git commit -m "add login form validation"

# Stage all tracked files AND commit in one step
git commit -am "fix typo in readme"
💡 Good commit messages

Write the message as a command: "add login form" not "added login form" or "login form stuff". Keep it under 72 characters. Describe what changed, not how.

git log — browse the history

bash
# Full log
git log

# One line per commit (most useful)
git log --oneline
a3f8b12 add login form validation
9c1e405 fix typo in readme
2b77d3a initial commit

# Visual branch graph
git log --oneline --graph --all

┌────────────────────────────────────────┐ │ your normal working day │ └────────────────────────────────────────┘ │ ▼ ┌─────────────────┐ │ edit files │ ← write code └────────┬────────┘ │ ▼ ┌─────────────────┐ │ git status │ ← see what changed └────────┬────────┘ │ ▼ ┌─────────────────┐ │ git add . │ ← stage the changes └────────┬────────┘ │ ▼ ┌─────────────────┐ │ git commit -m │ ← save the snapshot └────────┬────────┘ │ ▼ ┌─────────────────┐ │ git push │ ← sync to GitHub └────────┬────────┘ │ └──────────────▶ repeat

Working with GitHub

Connect your local repo to GitHub

  1. Create an empty repo on GitHub (don't add a README — you have one locally)
  2. Copy the HTTPS URL it gives you
  3. Run these commands in your project folder:
bash
# Tell Git where GitHub is
git remote add origin https://github.com/you/your-repo.git

# First push — -u sets the tracking relationship
git push -u origin main

# Every push after that — just this
git push

Getting changes from GitHub

bash
# Download + merge in one step (what you usually want)
git pull

# Download only — see what changed before merging
git fetch

fetch vs pull — what's the difference?

CommandWhat it doesUse when
git fetchDownloads changes from GitHub into a temporary area. Does not touch your working files.You want to see what others committed before merging it in
git pullDoes git fetch then immediately merges the result into your current branch.You trust the changes and just want to sync up (most of the time)
ℹ Note

git pull = git fetch + git merge. If you're on a shared branch and want to keep a clean history, use git pull --rebase instead.

Branches

A branch is a parallel version of your project. Think of main as the "official" version. When you want to add a feature or fix a bug, you create a branch — a separate timeline where you can make changes without affecting main. When you're happy, you merge it back.

Branch commands

bash
# List all local branches (* = current)
git branch
* main
  feature/login

# Create a new branch
git branch feature/login

# Switch to a branch
git switch feature/login
# or the older syntax:
git checkout feature/login

# Create AND switch in one step (most common)
git switch -c feature/login
# or:
git checkout -b feature/login

# Merge a branch into the current one
git switch main
git merge feature/login

# Delete a branch (after merging)
git branch -d feature/login

# Delete a branch on GitHub too
git push origin --delete feature/login

Feature branch workflow

branch diagram
main ──●──────────────────────────────────────●──▶ │ git switch -c feature/x ▲ │ │ feature/x └──●──────●──────●──────●──────────────┘ c1 c2 c3 c4 git merge feature/x Steps: 1. git switch -c feature/x ← branch off main 2. make commits on the branch 3. git switch main ← go back 4. git merge feature/x ← bring it in 5. git branch -d feature/x ← clean up

Undoing things

This is where most beginners panic. Stay calm — Git almost never permanently deletes anything. Here are your options, from safest to most drastic.

Discard unstaged changes to a file

Throws away edits you haven't staged yet. The file goes back to the last commit.

bash
git restore src/app.tsx    # discard edits
git restore .              # discard ALL unstaged edits

Unstage a file

You ran git add but changed your mind. This moves the file back to unstaged — your edits are NOT lost.

bash
git restore --staged src/app.tsx

Fix the last commit message

bash
git commit --amend -m "correct message"
⚠ Caution

Don't amend a commit you've already pushed to GitHub — it rewrites history and will cause problems for anyone who pulled it.

Undo a commit safely — git revert

Creates a new commit that undoes the changes from an old one. History is preserved — this is the safe way to undo public commits.

bash
# Find the commit hash
git log --oneline
a3f8b12 add broken feature
9c1e405 previous good commit

# Revert it
git revert a3f8b12

Erase the last commit — git reset

⛔ Danger — destructive

git reset --hard permanently deletes commits and all their changes. There is no undo. Never use this on commits you've already pushed.

bash
# Erase last commit, keep changes as unstaged
git reset HEAD~1

# NUCLEAR: erase last commit AND throw away all changes
git reset --hard HEAD~1

# Go back 3 commits
git reset --hard HEAD~3

Which undo do I use?

SituationCommandSafe?
Discard edits to a file (not staged) git restore <file> ✓ Safe — only affects that file
Unstage a file git restore --staged <file> ✓ Safe — edits still exist
Fix last commit message (not pushed) git commit --amend ✓ Safe locally
Undo a pushed commit git revert <hash> ✓ Safe — adds a new commit
Undo last commit, keep the changes git reset HEAD~1 ⚠ OK locally, don't push
Erase commits and all their changes git reset --hard HEAD~N ✗ Destructive — no undo

Advanced

◐ Intermediate–Advanced

git stash — shelve work temporarily

You're mid-feature but need to switch branches urgently. Stash saves your uncommitted changes off to the side.

bash
git stash             # save current work
git stash pop         # restore it
git stash list        # see all stashes
git stash drop        # discard the stash

git diff — see exact changes

bash
git diff              # unstaged changes
git diff --staged     # staged changes (what you're about to commit)
git diff main..feature # difference between two branches

Resolving merge conflicts

A conflict happens when two branches changed the same line differently. Git can't decide which to keep — it asks you.

  1. Run git merge feature/x — Git stops and tells you there's a conflict
  2. Open the conflicted file. Look for the markers Git inserted:
conflict markers
<<<<<<< HEAD
const greeting = "hello";    ← your version (current branch)
=======
const greeting = "hi";       ← their version (branch being merged)
>>>>>>> feature/x
  1. Edit the file to keep what you want — delete the markers and the version you don't want
  2. Stage the resolved file: git add src/app.tsx
  3. Complete the merge: git commit
💡 Tip

Use git status during a conflict to see exactly which files need resolving. Files marked both modified need your attention.

git rebase — replay commits on a new base

Rebase moves your branch's commits to start from a different point. The result is a linear, clean history — as if you had branched from the latest main.

merge vs rebase
── MERGE ────────────────────────────────────────── main A──B──C───────────────M ← merge commit └──D──E──F────┘ ── REBASE ───────────────────────────────────────── main A──B──C──D'──E'──F' ← linear history (D,E,F replayed on top of C) 
bash
# While on your feature branch:
git rebase main   # replay your commits on top of latest main
⚠ Rebase rule

Never rebase commits that have been pushed to a shared branch. Rebase rewrites history — if others have your old commits, their repos will diverge. Rebase is safe on your private feature branches.

git cherry-pick — grab one commit

Pull a single specific commit from any branch into the current one. Useful for hotfixes.

bash
# Find the commit hash
git log --oneline feature/x
a3f8b12 fix critical bug

# Apply just that commit to current branch
git cherry-pick a3f8b12

git tag — mark a release

bash
# Create a tag on the current commit
git tag v1.0.0

# Annotated tag with a message
git tag -a v1.0.0 -m "Initial release"

# Push tags to GitHub (doesn't happen automatically)
git push --tags

# List all tags
git tag

Common errors and fixes

detached HEAD

HEAD detached at a3f8b12

You checked out a specific commit hash instead of a branch name. Any commits you make here will be orphaned when you switch branches. Fix: git switch -c my-temp-branch to save your work, or git switch main to abandon it.

rejected push

error: failed to push some refs to 'origin/main' hint: Updates were rejected because the remote contains work that you do not have locally.

Someone else pushed commits while you were working. Fix: git pull first, resolve any conflicts, then push again.

merge conflict markers left in file

CONFLICT (content): Merge conflict in src/app.tsx Automatic merge failed; fix conflicts and then commit the result.

Open the file, find the <<<<<<< / ======= / >>>>>>> markers, edit to the correct state, then git add <file> and git commit.

accidentally committed to main

bash
# Move your commit to a new branch instead
git switch -c feature/my-work  # create branch with your commit
git switch main
git reset HEAD~1                # remove it from main
